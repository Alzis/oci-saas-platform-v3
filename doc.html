1. O Fluxo do Usuário

O usuário acessa a URL do frontend, que é um arquivo HTML servido publicamente por um Bucket no OCI Object Storage. O navegador carrega a aplicação (SPA), que por sua vez faz chamadas para o backend (api.dominio.com).

2. Rede e Balanceamento

As requisições para a API chegam a um Load Balancer da OCI. Ele atua como ponto de entrada central e estável, direcionando o tráfego para a VM de aplicação. Essa peça é fundamental para permitir o crescimento futuro, possibilitando adicionar mais VMs de aplicação sem alterar a arquitetura.

3. Ambiente de Execução

A arquitetura utiliza duas VMs VM.Standard.E2.1.Micro (Always Free): uma para a Aplicação (API, Proxy, DB, Cache) e outra dedicada à Observabilidade (Métricas e Dashboards), garantindo isolamento de recursos e simplicidade operacional.

4. Persistência de Dados

O banco de dados principal é o OCI MySQL HeatWave, um serviço gerenciado que oferece alta performance, backups automáticos e escalabilidade. O cache de sessão e de operações rápidas é feito pelo Redis, que roda como um container Docker na VM da aplicação.

5. Provisionamento (IaC)

Toda a infraestrutura na OCI (VCN, Subnet, VMs, LB, NSGs) é criada e gerenciada de forma automatizada pelo Terraform. Isso garante que o ambiente seja consistente, replicável e versionado, com baixo risco de falhas de provisionamento.

6. Deploy Contínuo (CI/CD)

Quando um desenvolvedor envia código novo para o GitHub, um processo de CI/CD (GitHub Actions) é acionado. Ele se conecta via SSH às VMs, baixa a versão mais recente do código e executa `docker compose up` para atualizar a aplicação no ar, sem intervenção manual.

Considerações Arquiteturais

O Papel Estratégico do Load Balancer

Mesmo com apenas uma VM de aplicação, a inclusão do OCI Load Balancer é uma decisão estratégica. Ele serve como um ponto de entrada fixo e desacoplado para a API. Quando a demanda crescer, podemos adicionar mais VMs de aplicação (escalabilidade horizontal) e registrá-las no Load Balancer sem nenhuma alteração na arquitetura ou no DNS. É a peça chave que prepara a plataforma para um crescimento gradual e sem atritos.

Banco de Dados Gerenciado: O Melhor dos Dois Mundos

Utilizar o OCI MySQL HeatWave é uma decisão pragmática. Delegamos a complexidade de gerenciamento, backup, patching e alta disponibilidade do banco de dados para a OCI, enquanto mantemos o restante da nossa infraestrutura enxuta e de baixo custo. A comunicação entre a API e o banco ocorre em uma rede privada, garantindo segurança e baixa latência.

Caminho de Evolução

Esta arquitetura é o ponto de partida. À medida que o SaaS cresce e gera receita, a evolução pode ocorrer de forma incremental:

  • Escalar VMs: Aumentar o número de VMs de aplicação (`api_instance_count` no Terraform).
  • Escalar o Banco: Aumentar a capacidade do MySQL HeatWave com poucos cliques no console da OCI ou via Terraform.
  • Externalizar o Cache: Migrar o container Redis local para um serviço gerenciado como o OCI Cache with Redis para maior resiliência.

Pilar 1: Infraestrutura como Código (IaC)

Terraform

Define toda a nossa nuvem em arquivos de código, permitindo criar, modificar e versionar a infraestrutura de forma segura e eficiente.

Consistência

Ambientes de dev e prod idênticos.

Automação

Criação da stack com um comando.

Replicabilidade

Sobe uma cópia exata em qualquer região.

Pilar 2: Containerização

Docker

Cada parte da aplicação (API, Cache, etc) é empacotada em um "container", isolando os serviços e garantindo que funcionem da mesma forma em qualquer lugar.

Isolamento

Dependências de um serviço não afetam outros.

Portabilidade

Roda igual no PC do dev e na nuvem.

Leveza

Mais eficientes que máquinas virtuais completas.

Pilar 3: Orquestração e Roteamento

Docker Compose & Traefik

Docker Compose define e executa os serviços da aplicação (API, Redis) com um único arquivo de configuração.

Traefik atua como nosso "porteiro" inteligente: gerencia o tráfego de entrada e obtém certificados SSL (HTTPS) automaticamente.

Pilar 4: Observabilidade Completa

Prometheus & Grafana

Não basta a aplicação funcionar, precisamos saber como ela está funcionando.

Prometheus

Coleta métricas de performance (CPU, RAM, etc).

Grafana

Cria dashboards visuais e alertas com os dados.

"Se não pode ser medido, não pode ser gerenciado."

Pilar 5: Deploy Automatizado

CI/CD com GitHub Actions

O processo de colocar uma nova versão da aplicação no ar é totalmente automatizado.

Git Push
→
GitHub Actions
→
SSH & Deploy

Resultado: Deploys rápidos, seguros e sem intervenção manual.

Conclusão

Esta plataforma demonstra uma arquitetura pragmática, econômica e sustentável. Ela oferece uma base sólida e simples para um MVP, com um caminho claro para crescimento gradual e escalabilidade, alinhada com as melhores práticas de DevOps.
